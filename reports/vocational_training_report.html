Abstract

Flight delays impose significant economic and operational costs on airlines, airports, and passengers worldwide. While airlines monitor delays internally, passengers and operational planners lack accessible, real-time predictive tools. This project builds an end-to-end machine learning system that predicts whether a commercial flight will arrive 15 or more minutes late, using only information available 24 hours before scheduled departure.

The system employs a LightGBM gradient-boosted decision tree trained on the U.S. Bureau of Transportation Statistics (BTS) On-Time Performance dataset. A 17-feature engineering pipeline transforms raw flight records through cyclic temporal encoding, Haversine great-circle distance computation, K-Fold smoothed target encoding, historical delay aggregates, and holiday proximity detection. A strict temporal split (January–August for training, September–October for validation, November–December for testing) simulates real deployment conditions where the model only ever predicts genuinely future flights.

Evaluation uses Precision–Recall AUC (PR-AUC) as the primary metric—more informative than ROC-AUC given the ~20% imbalanced positive class. SHAP TreeExplainer provides per-prediction feature attributions, enabling interpretable risk explanations. A comprehensive three-stage leakage audit (blacklist enforcement, high-correlation scan, and SHAP importance sanity check) ensures that no post-departure information contaminates the predictions.

The trained model is deployed as a production-ready web application: a FastAPI backend serves predictions with SHAP-based explanations via a REST API, and a React + Tailwind CSS frontend presents an interactive flight lookup interface with a probability gauge, risk-factor bar chart, and dark-mode support. Both services are containerised with Docker Compose for reproducible, single-command deployment. This project demonstrates that principled machine learning—combining rigorous leakage prevention, appropriate evaluation metrics, model interpretability, and modern full-stack engineering—can produce actionable predictive tools for transportation logistics.

1. Introduction

Flight delays are one of the most persistent operational challenges in commercial aviation. According to the U.S. Federal Aviation Administration (FAA), flight delays cost the U.S. economy over $30 billion annually, encompassing airline operational costs, passenger lost productivity, and downstream cascading effects across airport networks. In 2023, approximately 21% of U.S. domestic flights arrived more than 15 minutes behind schedule, affecting hundreds of millions of passengers.

The causes of flight delays are multifaceted: weather disruptions, air traffic congestion, mechanical issues, crew scheduling conflicts, and airport capacity constraints all contribute. While airlines maintain internal operations centres that monitor delays, the travelling public and third-party operational planners have limited access to predictive delay information. Existing consumer-facing tools (e.g., FlightAware, Google Flights) primarily report real-time status rather than pre-departure predictions.

From a machine learning perspective, flight delay prediction is a compelling problem. It features a moderately imbalanced binary target (~20% delayed), rich categorical features (airline, airport codes, aircraft tail numbers), temporal patterns (hour-of-day, day-of-week, seasonal effects), and geospatial structure (route distances). The problem also introduces a critical constraint not found in typical classification tasks: temporal information leakage. Many columns in flight performance datasets—such as taxi times, actual departure and arrival times, and carrier delay breakdowns—are only available after a flight has departed. Using these features would produce artificially inflated accuracy that is entirely useless in practice.

This project addresses this challenge by building a complete machine learning pipeline with a strict T-24h (24 hours before departure) information boundary. Every feature used by the model—from cyclic departure-time encoding to smoothed historical airline delay rates—is knowable a full day before the flight departs. The system spans the full ML lifecycle: data ingestion, temporal splitting, feature engineering, model training with Bayesian hyperparameter optimisation, evaluation with SHAP interpretability, and deployment as a full-stack web application with Docker containerisation.

2. Background Study

2.1 The Flight Delay Problem—Operational Context

Commercial aviation operates on tightly coupled networks. A single delayed departure at a hub airport can cascade through connecting flights, crew rotations, and gate assignments, creating system-wide disruptions. The Bureau of Transportation Statistics (BTS) tracks on-time performance for all U.S. domestic flights operated by carriers with at least 0.5% of domestic scheduled-service passenger revenue.

BTS defines a flight as “delayed” when it arrives 15 or more minutes after its scheduled arrival time. This 15-minute threshold is the industry standard used by the Department of Transportation (DOT) for regulatory reporting and forms the basis for contractual obligations between airlines and airports.

Delay causes are classified into five categories: carrier delays (maintenance, crew, aircraft cleaning), weather delays, National Aviation System (NAS) delays (air traffic control, airport operations), security delays, and late aircraft delays (propagation from a previous flight). Notably, these causal breakdowns are only available post-flight and must not be used for prediction.

2.2 Traditional Approaches and Their Limitations

Traditional delay prediction relies on operations research methods: queuing models for airport congestion, network flow models for propagation effects, and statistical regression on historical averages. Airlines use proprietary systems that combine real-time weather feeds, air traffic flow management data, and crew scheduling information.

These approaches have significant limitations for general-purpose prediction. Queuing models require real-time airport state information that is unavailable 24 hours in advance. Statistical averages fail to capture non-linear interactions between factors (e.g., a particular airline at a particular airport during holiday season). Proprietary airline systems are not accessible to passengers or third-party planners.

2.3 Machine Learning for Flight Delay Prediction

Machine learning offers a data-driven alternative that can capture complex, non-linear relationships from historical flight records. Gradient-boosted decision trees (GBDT), particularly implementations such as LightGBM and XGBoost, have emerged as the dominant approach for tabular prediction tasks. They handle mixed feature types natively, are robust to feature scaling, provide built-in feature importance, and train efficiently on datasets with millions of records.

A critical challenge unique to this domain is information leakage. Flight performance datasets contain both pre-departure features (scheduled times, airline, airports) and post-departure features (actual times, taxi durations, delay breakdowns). Naïve inclusion of post-departure features produces models with artificially high accuracy (>90%) that are entirely useless in practice because those features are unknown at prediction time. Rigorous feature blacklisting and leakage auditing are therefore essential.

2.4 SHAP Interpretability

Shapley Additive exPlanations (SHAP), introduced by Lundberg and Lee (2017), provide a theoretically grounded framework for explaining individual predictions. SHAP values decompose each prediction into additive contributions from each feature, satisfying desirable axiomatic properties: local accuracy, missingness, and consistency. For tree-based models, the TreeSHAP algorithm computes exact Shapley values in polynomial time, making it practical for production use.

In the flight delay context, SHAP enables actionable explanations such as: “This flight has a 72% delay probability primarily because the airline’s historical delay rate is high (+15%) and the departure time is during peak evening hours (+12%).”

3. Problem Statement

Despite extensive data collection by the Bureau of Transportation Statistics, extracting actionable pre-departure delay predictions remains challenging. Existing prediction systems either require real-time operational data unavailable to the public, or fail to enforce the temporal boundary between pre- and post-departure information, thereby producing misleading performance estimates.

This project addresses the following problem:

Given only information available 24 hours before a scheduled departure—airline, origin airport, destination airport, scheduled departure time, and aircraft tail number—can a machine learning model accurately predict whether the flight will arrive 15 or more minutes late?

The task is formulated as a supervised binary classification problem:

  • Label: y = 1 if arrival delay ≥ 15 minutes; y = 0 otherwise
  • Positive class (~20%): Delayed flights (minority class)
  • Negative class (~80%): On-time flights (majority class)

Beyond raw accuracy, the model must satisfy four domain-specific requirements:

  1. Strict temporal leakage prevention: A blacklist of 14 post-departure columns (26 including case variants) is enforced at data loading time. A SHAP-based sanity check further verifies that no single feature contributes more than 40% of total importance.
  2. Appropriate evaluation for class imbalance: With a ~20% positive rate, accuracy alone is misleading (a naïve “always on-time” classifier achieves ~80%). PR-AUC, which focuses on the minority class, serves as the primary evaluation metric.
  3. Interpretable predictions: Each prediction must include the top contributing factors with signed SHAP impact values, enabling users to understand why a flight is predicted to be delayed.
  4. Temporal evaluation integrity: Training, validation, and test sets must be split chronologically—not randomly—to simulate real deployment where the model predicts future flights based on past data only.

4. Objectives

Primary Objective

To develop, evaluate, and deploy a machine learning binary classification system that predicts flight arrival delays ≥15 minutes using only pre-departure features, achieving a PR-AUC greater than 0.35 on a temporally held-out test set, with zero data leakage and per-prediction SHAP explanations.

Secondary Objectives

  1. Data Pipeline: Build a robust data loading module that enforces a blacklist of 14 post-departure columns (26 including case variants), preventing temporal information leakage at the earliest stage.
  2. Temporal Splitting: Implement a calendar-based chronological split (Jan–Aug / Sep–Oct / Nov–Dec) with assertion guards that raise an error if date ranges overlap.
  3. Feature Engineering: Design a 17-feature pipeline comprising cyclic sin/cos encoding, Haversine great-circle distance (using 139 airport coordinates), K-Fold smoothed target encoding (λ=20), historical delay rate aggregates (with minimum-sample thresholds of 30 and 100), and US federal holiday proximity flags (±2 days).
  4. Model Training: Train a LightGBM gradient-boosted decision tree with up to 3,000 boosting rounds, early stopping (patience = 100), and optional class-weight balancing.
  5. Hyperparameter Optimisation: Implement Optuna-based Bayesian optimisation (TPE sampler) over 9 hyperparameters with 50 trials, maximising validation PR-AUC.
  6. Comprehensive Evaluation: Report PR-AUC, ROC-AUC, F1-score, precision at 50% recall, recall at 80% precision, Brier score, and confusion matrix on the held-out November–December test set.
  7. SHAP Interpretability: Generate global feature importance (beeswarm and bar plots), per-prediction top-5 factor explanations, and partial dependence plots for the top 3 features.
  8. Leakage Audit: Implement a three-stage audit—blacklist check, high-correlation scan (|r| > 0.95 threshold), and SHAP importance sanity check (>40% single-feature threshold)—run on every training cycle.
  9. REST API: Deploy the model as a FastAPI backend with POST /predict and GET /health endpoints, Pydantic v2 request validation, CORS support, and request-logging middleware.
  10. Interactive Frontend: Build a React 18 + Vite + Tailwind CSS single-page application with a flight input form, probability gauge meter (Recharts), SHAP factor chart, loading skeleton, error toast, and dark mode.
  11. Docker Deployment: Containerise both backend and frontend with Docker Compose, including health checks, read-only artifact volumes, and a shared bridge network.

5. Literature Review

Research on flight delay prediction has evolved significantly over the past two decades, progressing from simple statistical models to sophisticated ensemble methods and interpretable AI.

Rebollo and Balakrishnan (2014) applied Random Forests to predict departure delays at major U.S. airports, demonstrating that tree-based ensembles outperform linear models by capturing non-linear interactions between weather, traffic, and temporal features. Their work highlighted the importance of temporal features (hour-of-day, day-of-week) as primary predictors of delay.

Mueller and Chatterji (2002) established the foundational understanding of delay propagation in air traffic networks, showing that delays cascade through connected flights in a hub-and-spoke system. This insight motivated the inclusion of historical route-level and airline-level delay rates as features in the present work.

Kim et al. (2016) compared multiple classification algorithms—Logistic Regression, SVM, Random Forest, and Gradient Boosting—on BTS data and found that Gradient Boosting consistently achieved the highest AUC scores. Critically, their work also emphasised the importance of removing post-departure features to avoid information leakage.

Ke et al. (2017) introduced LightGBM, a gradient-boosting framework that uses histogram-based splitting and leaf-wise tree growth. LightGBM achieves state-of-the-art performance on tabular data while training 10–20× faster than traditional GBDT implementations, making it practical for large-scale flight datasets containing millions of records.

Akiba et al. (2019) introduced Optuna, a hyperparameter optimisation framework using Tree-structured Parzen Estimators (TPE). TPE models the conditional probability of promising hyperparameter configurations and samples preferentially from high-performing regions, converging to near-optimal configurations in 30–50 trials on typical 10-dimensional search spaces.

Lundberg and Lee (2017) introduced SHAP (SHapley Additive exPlanations), providing a unified framework for feature attribution grounded in cooperative game theory. For tree-based models, their TreeSHAP algorithm computes exact Shapley values in polynomial time, enabling per-prediction explanations at production speed.

Choi et al. (2016) demonstrated that feature engineering—specifically cyclic encoding of temporal variables—significantly improves delay prediction accuracy. Cyclic sin/cos encoding preserves the circular nature of time (i.e., 23:59 is temporally adjacent to 00:01), which standard ordinal encoding misrepresents.

Hensman and Masko (2015) investigated temporal train/test splitting for time-dependent prediction tasks, demonstrating that random shuffled splits produce optimistically biased performance estimates. For flight delay prediction, where patterns evolve seasonally and are influenced by non-stationary factors, temporal splitting is essential for honest model evaluation.

7. Data Preprocessing

7.1 Loading and Cleaning

The load_raw() function in src/data/loader.py handles initial data ingestion in six steps:

  1. Format detection: Accepts both CSV and Parquet formats. BTS data is typically downloaded as CSV; Parquet is used for intermediate cached files.
  2. Column standardisation: BTS column names vary between download sessions (e.g., ARR_DELAY vs. ArrDelay). The loader maps 10 known variants to canonical names automatically.
  3. Blacklist enforcement: All 26 blacklisted column strings (14 unique columns × 2 case variants) are immediately dropped. A warning is logged if any were present.
  4. Target creation: The binary label is created as y = (ArrDelay ≥ 15).astype(np.int8). If the BTS pre-computed indicator column ArrDel15 is available, it is used directly.
  5. Cancelled/diverted removal: Rows with null ArrDelay (cancelled or diverted flights) are dropped, as they have no meaningful delay to predict.
  6. Chronological sorting: The output DataFrame is sorted by FL_DATE for downstream temporal splitting.

8. Temporal Train–Validation–Test Splitting

8.1 Rationale

Unlike datasets with independent, identically distributed observations, flight delay data is inherently temporal. A random shuffled split would allow the model to “see” December flights during training and then be “tested” on July flights—a scenario that never occurs in real deployment, where the model must predict future flights from past data only. Temporal splitting simulates this real-world deployment condition.

8.2 Split Strategy

SplitMonthsApprox. SizePurpose
TrainJanuary–August (months 1–8)~70%Model learning
ValidationSeptember–October (months 9–10)~15%Hyperparameter tuning, threshold selection
TestNovember–December (months 11–12)~15%Final held-out evaluation

8.3 Integrity Guards

The temporal_split() function enforces strict chronological ordering with runtime assertions:

assert max(train_dates) < min(val_dates),   "Train/val date overlap detected"
assert max(val_dates)   < min(test_dates),  "Val/test date overlap detected"

If these assertions fail (e.g., due to multi-year data without filtering), a ValueError is raised, preventing subtle leakage from overlapping date ranges.

8.4 Statistics Reporting

The get_split_stats() function returns per-split statistics—row count, number delayed, number on-time, and delay rate—enabling verification that class distributions remain consistent across splits.

10. Feature Engineering

Feature engineering transforms raw flight record columns into 17 predictive features, implemented in src/features/pipeline.py. The FeaturePipeline class orchestrates five sequential stages.

10.1 Stage 1—Cyclic Temporal Encoding

Raw time variables (departure time, day-of-week, month) are inherently circular: 23:59 is temporally adjacent to 00:01, but numerically distant (1439 vs. 0 in minutes). Standard ordinal encoding misrepresents this relationship. Sin/cos encoding maps each value to a point on the unit circle, preserving circular proximity:

xsin = sin(2π · t / T)       xcos = cos(2π · t / T)
Output FeaturesInput VariablePeriod (T)
CRS_DEP_TIME_sin, CRS_DEP_TIME_cosMinutes since midnight (0–1439)1440
DayOfWeek_sin, DayOfWeek_cosDay of week (1–7)7
Month_sin, Month_cosMonth (1–12)12
departure_hourHour extracted (0–23)— (ordinal)

Implementation detail: HHMM integers are converted to minutes since midnight via hours = hhmm // 100, minutes = hhmm % 100, total = hours × 60 + minutes. Out-of-range values are clamped to [0, 1439].

Refer to notebook 02_feature_engineering.ipynb for cyclic encoding scatter plots (departure time, day of week, month).

10.2 Stage 2—Haversine Great-Circle Route Distance

Route distance captures the physical length of the flight path, which correlates with delay patterns. The Haversine formula computes the great-circle distance between two points on a sphere:

a = sin²(Δlat / 2) + cos(lat1) · cos(lat2) · sin²(Δlon / 2)
distance = 2 × R × arcsin(√a)     where R = 6,371.0088 km

The implementation is fully vectorised using NumPy for performance on millions of rows. Airport coordinates for 139 major U.S. airports (e.g., ATL, LAX, ORD, DFW, JFK) are stored in a lookup dictionary within src/features/geospatial.py.

Fallback strategy:

  1. If both airports have known coordinates → Haversine distance in kilometres
  2. If one or both coordinates are missing → DISTANCE (statute miles) × 1.60934
  3. If neither is available → NaN (handled natively by LightGBM)
Refer to notebook 02_feature_engineering.ipynb for route distance distribution histogram.

10.3 Stage 3—Historical Delay Rate Aggregates

Historical delay rates capture systemic patterns: certain routes and airlines consistently experience higher delay rates due to structural factors (hub congestion, equipment allocation, geographic weather exposure).

Route Delay Rate (minimum 30 flights)

w = min(n, 30) / 30      route_delay_rate = w · (ndelayed + 1) / (ntotal + 2) + (1 − w) · pglobal

Airline Delay Rate (minimum 100 flights)

w = min(n, 100) / 100      airline_delay_rate = w · (ndelayed + 1) / (ntotal + 2) + (1 − w) · pglobal

Routes or airlines with fewer flights than the minimum threshold are smoothed toward the global average via the blending weight w. Laplace smoothing (+1 / +2) prevents zero/one extreme rates. Both rates are computed on training data only and applied to validation/test sets via lookup, preventing target leakage.

10.4 Stage 4—K-Fold Smoothed Target Encoding

High-cardinality categorical features—ORIGIN (300+ airports), DEST (300+ airports), TAIL_NUM (5,000+ aircraft)—cannot be one-hot encoded without extreme dimensionality. Target encoding maps each category to the mean target value (delay rate) of that category, producing a single informative numeric feature per column.

Leakage prevention via K-Fold out-of-fold encoding:

  1. Split training data into K=5 folds (shuffle=False to preserve temporal order)
  2. For each fold: compute category means on the remaining 4 folds; apply to the held-out fold
  3. Each training row’s encoding is derived only from data that does not include itself

Smoothing regularisation:

TE(category) = (n · pcategory + λ · pglobal) / (n + λ)     where λ = 20.0

Categories with small sample sizes (n << λ) shrink toward the global mean, preventing overfitting on rare airports or aircraft. Unseen categories at inference time receive the global mean as a fallback.

Columns encoded: OP_CARRIER → OP_CARRIER_te, ORIGIN → ORIGIN_te, DEST → DEST_te, TAIL_NUM → TAIL_NUM_te.

10.5 Stage 5—Holiday Proximity

Flights near major US holidays experience elevated delay rates due to increased passenger volume, seasonal weather, and operational strain. Ten US federal holidays are detected (4 fixed-date + 6 variable-date):

  • Fixed: New Year’s Day (1 Jan), Independence Day (4 Jul), Veterans Day (11 Nov), Christmas Day (25 Dec)
  • Variable: MLK Day (3rd Monday in Jan), Presidents’ Day (3rd Monday in Feb), Memorial Day (last Monday in May), Labor Day (1st Monday in Sep), Columbus Day (2nd Monday in Oct), Thanksgiving (4th Thursday in Nov)
is_near_holiday = 1  if min(|flight_date − holiday_date|) ≤ 2 days,  else 0

Implementation uses chunked vectorised broadcasting (50,000 rows per chunk) for memory efficiency on large datasets.

11.2 Algorithm Overview

LightGBM (Light Gradient Boosting Machine), developed by Microsoft Research (Ke et al., 2017), is an ensemble of decision trees trained sequentially. Each new tree is fitted on the negative gradient (pseudo-residuals) of the binary cross-entropy loss with respect to the current ensemble’s predictions.

Key algorithmic innovations:

  1. Histogram-based splitting: Bins continuous features into 255 discrete buckets, reducing split-finding complexity from O(n × features) to O(bins × features)
  2. Leaf-wise (best-first) growth: Expands the leaf with the maximum delta loss, producing deeper and more accurate trees with fewer total splits compared to level-wise growth
  3. Gradient-based One-Side Sampling (GOSS): Retains all high-gradient (poorly predicted) samples while sub-sampling low-gradient samples, accelerating training without significant accuracy loss
  4. Exclusive Feature Bundling (EFB): Bundles mutually exclusive sparse features to reduce the effective number of features

12.2 Optimisation Protocol

  • Sampler: TPESampler(seed=42)—Tree-structured Parzen Estimator
  • Direction: Maximise
  • Objective metric: PR-AUC (Average Precision Score) on the validation set
  • Per-trial training: Up to 2,000 boosting rounds with early stopping (patience = 50)
  • Default budget: 50 trials
  • Output: best_params.json saved to the artifacts directory with the best PR-AUC score and corresponding hyperparameters

12.3 Why TPE over Grid or Random Search?

TPE models the probability distribution of “good” hyperparameter configurations and samples preferentially from promising regions of the search space. On a 9-dimensional search space like the one above, TPE typically converges to near-optimal configurations in 30–50 trials, whereas random search may require hundreds to achieve comparable results.

13.3 Threshold Selection

Rather than using the default 0.5 threshold, the optimal threshold is selected by maximising F1-score on the validation set:

F1(t) = 2 · precision(t) · recall(t) / (precision(t) + recall(t))
optimal_threshold = argmaxt F1(t)

This allows tuning the precision–recall trade-off: a lower threshold catches more delays (higher recall) but generates more false alarms (lower precision).

14. Model Evaluation Metrics

14.1 Why Not Just Accuracy?

With a ~20% positive rate, a trivial “always predict on-time” classifier achieves ~80% accuracy. This makes accuracy a poor discriminator between useful models and useless baselines. Metrics that focus on the minority class (delayed flights) provide more meaningful comparisons.

14.2 Primary Metric: PR-AUC

PR-AUC (Precision–Recall Area Under Curve) summarises the precision–recall trade-off across all classification thresholds. Unlike ROC-AUC, which can be inflated by the large number of true negatives, PR-AUC is sensitive to the model’s ability to correctly identify delayed flights. A PR-AUC of 0.35 means the model significantly outperforms random chance (baseline ≈ 0.20, the positive class rate).

17.1 Error Types and Their Costs

  • False Positives (FP): On-time flights incorrectly predicted as delayed. Cost: unnecessary passenger anxiety, potential rebooking expenses. Moderate cost.
  • False Negatives (FN): Delayed flights incorrectly predicted as on-time. Cost: missed connections, lack of preparedness, downstream planning failures. Higher cost.

Given the asymmetric costs, the model should prioritise recall (minimising false negatives) while maintaining acceptable precision. The scale_pos_weight parameter and threshold tuning both provide mechanisms to shift the model toward higher recall when desired.

21. Design Decisions and Trade-offs

DecisionRationale
LightGBM over deep learningHandles mixed feature types natively, trains fast, and is directly compatible with SHAP TreeExplainer. Deep learning requires more data and offers less interpretability for tabular tasks.
PR-AUC as primary metricMore informative than ROC-AUC on the ~20% imbalanced positive class. Focuses evaluation on the minority (delayed) class.
Temporal split (no shuffle)Simulates real deployment where the model only sees future flights. Random splits produce optimistically biased estimates.
scale_pos_weight for imbalanceNative LightGBM parameter; avoids the distribution distortion caused by SMOTE.
K-Fold target encoding (λ=20)Prevents target leakage inherent in naïve mean encoding. Smoothing regularises rare categories.
Threshold tuning on validation setEnables operational flexibility: stakeholders can choose between catching more delays (high recall) and fewer false alarms (high precision).
14-column blacklist at load timeDefence-in-depth: leakage is blocked at the earliest possible stage, before any feature engineering occurs.
SHAP per predictionUsers need to understand why a flight is predicted delayed, not just the probability. Builds trust and enables informed decision-making.
Decoupled FastAPI + ReactBackend can serve mobile apps and third-party integrations independently. Frontend can be replaced without touching the model.
Docker ComposeSingle-command deployment with health checks and read-only artifact volumes. No model is baked into the Docker image.

22. Limitations

22.1 No Real-Time Weather Data

Weather is the single largest cause of flight delays, accounting for approximately 30% of all delay minutes. However, airport-specific weather predictions 24 hours in advance lack sufficient accuracy. The model captures weather effects only indirectly through seasonal patterns (month, holiday proximity) and historical route delay rates.

22.2 Single Data Source and Geography

The model is trained exclusively on U.S. domestic flights from BTS data. It does not generalise to international routes, non-U.S. carriers, or regional airports not covered by BTS reporting requirements.

22.3 Temporal Stationarity Assumption

The model assumes that delay patterns from the training period (January–August) generalise to the test period (November–December). In practice, airline schedules, airport infrastructure, and fleet compositions evolve over time. Annual retraining is necessary to maintain prediction quality.

22.4 No Real-Time Operational State

The model cannot account for the current state of the air traffic system: active ground stops, runway closures, ongoing cascading delays, or gate conflicts. These factors are only available minutes to hours before departure, not 24 hours in advance.

22.5 Target Encoding Cold Start

New airlines, airports, or aircraft not seen during training receive the global mean delay rate via the fallback mechanism. Predictions for these entities are reasonable but uninformed until sufficient data accumulates.

22.6 Binary Classification Simplification

The model predicts only “delayed” (≥15 min) vs. “on-time” (<15 min). It does not predict delay magnitude (e.g., 20 minutes vs. 3 hours), which would be valuable for connection planning. A regression formulation could address this at the cost of different evaluation challenges.

22.7 No Prospective Validation

Evaluation is limited to a held-out temporal split from the same calendar year. The model has not been prospectively validated on a subsequent year’s data, which is essential before any operational deployment.

23. Future Scope

23.1 Weather Feature Integration

Integrating NOAA weather forecasts (temperature, precipitation, wind speed, visibility) at origin and destination airports would significantly improve prediction accuracy. APIs such as the National Weather Service or OpenWeatherMap could provide this data.

23.2 Real-Time Delay Propagation

Incorporating the current delay state of the aircraft (is it already delayed on a previous leg?) and the origin airport (current average delay) would capture cascading effects. This shifts the prediction window from T-24h to T-2h, trading forecast horizon for accuracy.

23.3 Multi-Year Training

Training on 3–5 years of BTS data would improve robustness to seasonal variations and unusual weather years, and enable proper time-series cross-validation with multiple train/test windows.

23.4 Alternative GBDT Implementations

A systematic comparison with XGBoost and CatBoost could identify marginal improvements. CatBoost’s ordered target encoding is a particularly interesting alternative to the manual K-Fold approach used here.

23.5 Stacking Ensemble

Training a meta-learner (e.g., Logistic Regression) on out-of-fold predictions from multiple GBDT variants could improve performance beyond single-model tuning.

23.6 Regression Target (Delay Minutes)

Predicting the continuous delay magnitude rather than a binary classification would enable more nuanced decision-making. A quantile regression approach could additionally provide uncertainty estimates.

23.7 Streamlit Cloud Deployment (Completed)

The application has been deployed to Streamlit Community Cloud, providing free public access at https://flight-delay-predictor-y7veehaanvd2umrlc4z4zh.streamlit.app/. The Streamlit app reuses the same LightGBM model and feature pipeline in-process, features an interactive 3D globe (globe.gl), airline-branded boarding pass cards, a SHAP waterfall chart, and a modular component architecture (config.py, core.py, components/ package). Source code is hosted on GitHub at https://github.com/Paras-Dhatwalia/Flight-Delay-Predictor.

23.8 Mobile Application

Wrapping the REST API in a mobile application (React Native or Flutter) would make predictions accessible to passengers on the go, with push-notification support for updated predictions closer to departure.

24. Conclusion

This project demonstrates a complete end-to-end machine learning pipeline for flight delay prediction—from raw BTS data ingestion through feature engineering, model training, evaluation, and deployment as a production-ready full-stack web application.

The system addresses the critical challenge of temporal information leakage through multiple defence layers: a 14-column blacklist (26 case variants) enforced at data loading time, a temporal train/validation/test split with chronological assertion guards, K-Fold out-of-fold target encoding to prevent within-sample leakage, and a three-stage leakage audit combining blacklist verification, high-correlation scanning, and SHAP importance analysis.

The 17-feature engineering pipeline captures five distinct signal types—temporal (cyclic sin/cos encoding), spatial (Haversine great-circle distance using 139 airport coordinates), categorical (K-Fold smoothed target encoding with λ=20), historical (route and airline delay rate aggregates with minimum-sample thresholds), and calendar (US federal holiday proximity within ±2 days)—all derived exclusively from information available 24 hours before departure.

LightGBM was selected for its native handling of mixed feature types, built-in missing-value support, efficient histogram-based training, and compatibility with SHAP TreeExplainer for per-prediction interpretability. Optuna-based Bayesian hyperparameter optimisation (TPE sampler, 50 trials, PR-AUC objective) further improves model quality beyond default parameters.

Evaluation uses PR-AUC as the primary metric—appropriate for the ~20% imbalanced positive class—supplemented by ROC-AUC, F1, calibration (Brier score), precision at fixed recall, and recall at fixed precision. SHAP analysis provides both global feature importance (beeswarm and bar plots) and per-prediction explanations (top 5 factors with signed impacts).

The trained model is deployed through a FastAPI REST API with Pydantic v2 request validation, CORS middleware, and request logging, paired with a React + Tailwind CSS frontend featuring a colour-coded probability gauge, SHAP factor bar chart, and dark-mode support. Both services are containerised with Docker Compose for reproducible, single-command deployment with health checks and read-only artifact volumes.

More broadly, this project demonstrates that principled machine learning—combining rigorous leakage prevention, appropriate evaluation metrics for imbalanced data, model interpretability, and modern software engineering practices—can produce useful predictive tools for transportation logistics. With the addition of real-time weather data and delay propagation features, such systems could provide meaningful decision support for passengers, airlines, and airport operations planners.
